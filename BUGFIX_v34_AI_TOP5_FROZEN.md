# BUGFIX v3.4 — AI 지능형 추천 TOP5 고착(매일 동일) 문제 근본 수정

## 증상
- 매일 300건 이상 로그가 축적되는데도 AI 지능형 추천 1~5순위가 전혀 변하지 않음.

## 재현 검증 결과
- 기존 코드를 동일 로그로 반복 실행 → 배포 중인 보고서와 **동일한 1~4순위가 그대로 재현됨**
  (시드 없는 무작위 샘플링에도 불구하고 결과는 사실상 결정론적).
- 수정 후 일자별 신규 로그 300건 누적 시뮬레이션 → Day1↔Day2, Day2↔Day3 동일 조합 **0/5**
  (순위·조합이 매일 변동함을 확인).

## 근본 원인 (4가지)

| # | 위치 | 문제 |
|---|------|------|
| 1 | `ai_intelligent_analyzer.py` `_build_log_stats` | 조합 단위 중복 제거(`seen_keys`) → 매일 300건이 쌓여도 유사 조합 반복이면 통계가 전혀 변하지 않음 |
| 2 | `_l1_log_freq_hit` | "상위 15개 번호 **집합** 포함 여부"만 평가 → 누적될수록 top15 집합이 수렴(대수의 법칙)해 며칠 만에 고정 |
| 3 | `_composite_score` 구조 | StatScore(60%)는 lotto.xlsx 기반이라 주간 추첨 전까지 완전 고정. 고정 점수 함수의 argmax는 후보 풀이 무작위여도 같은 답 |
| 4 | `_collect` 내부 순서 | `is_intelligent`(자기 추천) 제외 체크가 카운팅 **이후**에 있어, AI 자신의 직전 추천이 로그 통계에 섞이는 자기 강화 루프 존재 |

추가로 `.github/workflows/lotto_scheduler.yml`에 `schedule` 트리거가 없어
외부 cron(cron-job.org)이 멈추면 Supabase의 추천 레코드가 영영 갱신되지 않는 단일 실패점이 있었음.

## 수정 내용

### 1. `ai_intelligent_analyzer.py` (v3.3 → v3.4)
- **[A] 최신성 가중 (Recency Weighting)**: 각 로그 레코드에 `w = 0.5^(경과일/2)` 가중 적용
  (반감기 2일, 상수 `RECENCY_HALF_LIFE`). 오늘 쌓인 로그가 과거 로그보다 항상 큰 영향력을
  가지므로 **매일 통계가 실제로 변동**함.
- **[B] 조합 중복 제거 폐지**: 같은 조합의 반복 생성도 신호로 간주하여 가중 빈도에 반영.
- **[C] L1 연속화**: 집합 포함 비율(이산) → 정규화 가중 빈도 평균(연속).
  빈도의 미세 변화가 점수에 즉시 반영됨.
- **[D] L3 신규성(Novelty) 도입**: `1 − (직전 추천과의 최대 중복 수)/6`.
  직전 `is_intelligent` TOP5를 cleanup **전에** 수집(`_collect_prev_intelligent`)하여
  어제와 동일한 조합이 또 1순위가 되는 것을 구조적으로 차단.
  - LogScore = L1×0.40 + L2×0.40 + L3×0.20 (기존: L1×0.50 + L2×0.50)
- **[E] 자기 강화 루프 차단**: `is_intelligent` 레코드를 빈도/동반출현 카운팅 및
  후보 풀(`_add_log_candidates`)에서 완전 제외.
- 보고서에 `오늘 신규 N건 / 최신 로그 가중 반감기 / 신규성(L3) 적용 여부` 표기 추가
  → 로그가 실제로 반영되는지 매일 눈으로 확인 가능.

### 2. `.github/workflows/lotto_scheduler.yml`
- `schedule` 트리거 추가:
  - 매일 `0 0 * * *` (00:00 UTC = **09:00 KST**) → daily 모드
  - 토요일 `30 13 * * 6` (13:30 UTC = **22:30 KST**) → saturday 모드
- 스케줄 트리거 시 KST 요일/시각 기준으로 모드 자동 판별 (기존 수동 실행은 그대로 유지).
- 외부 cron-job.org 의존 제거 → GitHub에 올리기만 하면 자동으로 매일 추천이 갱신됨.

## 변경 파일
- `ai_intelligent_analyzer.py` (핵심 수정)
- `.github/workflows/lotto_scheduler.yml` (자동 스케줄 추가)
- `BUGFIX_v34_AI_TOP5_FROZEN.md` (본 문서)

- 3일치 로그 누적 시뮬레이션: 일자별 TOP5 동일 조합 0/5 (변동 확인)
- `run_analysis()` 엔드투엔드 정상 동작 (보고서/로그/cleanup 모두 정상)

3. 1~5순위 조합이 전일 대비 달라지는지 확인 (L3 신규성에 의해 구조적으로 보장됨).

# 추가 수정 (v3.4.1) — 회차 불일치로 AI 추천이 대시보드에서 필터링되던 문제

## 증상
- GHA Actions는 성공, AI 분석도 성공("1227회차 is_intelligent 레코드 5건")인데
  대시보드는 1228회차 로그를 표시하면서 AI 추천이 보이지 않음 (일반 로그 폴백).

## 원인 (실제 데이터로 확인)
- 저장소 `lotto.xlsx`의 마지막 추첨일이 **2026-05-30(1226회차)** 로,
  6/6(토) 1227회차 추첨분이 미반영된 stale 상태.
- 기존 `get_round_context`의 회차 보정은 "토요일 21시 이후 또는 일요일"에만 +1 하는
  요일 기반 로직이라, **평일에 실행되면 1227로 잘못 계산**됨.
  (일요일에 실행된 로그는 1228로 기록 → 회차가 갈라짐)
- 대시보드는 로그의 최신 회차(1228)로 필터링하므로 1227로 저장된
  AI 추천(is_intelligent)이 화면에서 제외됨.

## 수정
1. **`log_utils.py` `get_round_context`** — 날짜 기반 회차 보정으로 교체:
   마지막 기록된 추첨일 이후 지난 '토요일 21:00 KST' 횟수만큼 회차 가산.
   → xlsx가 아무리 오래돼도 GHA/Streamlit/로컬 모두 동일 회차 계산 (1228 ✓).
   (부수 버그 수정: 기존 코드는 `df.iloc[0]`을 최신 행으로 가정했으나
    xlsx는 오름차순이라 2002년 1회차의 추첨일을 읽고 있었음)
2. **`gha_daily.py`** — 루틴 시작 시 `update_lotto.update_excel()` 호출로
   당첨번호 최신화 (실패해도 1번의 날짜 보정이 회차를 보장).
3. **`.github/workflows/lotto_scheduler.yml`** — `permissions: contents: write` 추가,
   갱신된 lotto.xlsx를 저장소에 자동 커밋하는 스텝 추가.
4. **`ai_intelligent_analyzer.py` `_load_rows`** — 회차 내림차순 정렬 보장:
   xlsx가 오름차순인데 rows[0]=최신을 가정해 Markov/gap 등 최신성 통계가
   뒤집혀 있던 잠재 버그 수정.

## 검증
- 보정 후 target_round = 1228 (stale xlsx 상태에서도 정확)
- 분석기 엔드투엔드 실행 → 1228회차 is_intelligent 레코드 15건 생성 확인
- 기존 테스트: 수정 전후 실패 목록 동일 (신규 실패 0건)
